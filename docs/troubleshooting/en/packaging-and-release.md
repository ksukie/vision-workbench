# Packaging and Release Troubleshooting

[Index](./README.md) | [Release Policy](../../legal/release_policy.md) | [中文](../zh-CN/打包与发布.md)

This page covers runtime version identity, update failures, wheel builds, `dist/`, release assets, and model files larger than 100 MB.

## Runtime Version And `pip show` Disagree

In editable mode, `pip show vision-workbench` displays distribution metadata generated by the last editable installation, while the launcher imports the live checkout. After changing branches or updating `pyproject.toml`, the two values can therefore differ without the launcher using an old copy. Use the in-app **Version Information** page to identify the running code. If the pip metadata must also match, close the application and refresh only the editable registration from the repository root:

```bash
python -m pip install --no-deps --force-reinstall --editable .
python -m pip show vision-workbench
```

Do not use this command to repair a formal wheel installation; reinstall the matching official wheel instead.

## Update Check Or One-Click Update Fails

An offline connection, GitHub rate limit, invalid release metadata, missing compatible asset, missing SHA-256, oversized asset, or changed runtime dependency contract must not be reported as “up to date.” Retry the explicit check later or open the official Release page from the version page. If one-click installation remains unavailable, follow that Release's manual installation instructions; do not bypass the hash or dependency-contract gate.

The updater verifies the download twice and applies it only after Qt exits. A Python update also verifies the installed wheel identity before restart. A Windows EXE is self-tested before replacement and keeps the previous executable beside it. On Windows, inspect `%LOCALAPPDATA%\VisionWorkbench\updates\<version>\update.log` for the exact failure. A failed `.part` file is safe to remove after all Vision Workbench processes have exited.

## Build Fails

Use:

```bash
python -m pip install build
python -m build
```

If isolated build dependencies cannot be downloaded, reuse the current environment:

```bash
python -m build --no-isolation
```

## dist Directory

`dist/` contains local build outputs. It must not be committed to the source repository.

## Release Asset Check Failed

The release checker blocks large model files in Git:

```bash
python scripts/check_release_assets.py
```

Model files larger than 100 MB should be moved to local download instructions, Git LFS, or GitHub Release assets before publishing.

## Wheel Is Not a Complete Runtime

The base wheel installs the first-party Python packages, required resources, and entry points, but it is not a complete copy of the Git repository. Tests, development scripts, the complete vendored source tree, large model weights, and optional deep-learning dependencies may be excluded.

Use the matching Git tag or source archive when complete project source is required. For models, follow the relevant module documentation and use the built-in download action, a local model directory, or GitHub Release Assets.

A Python sdist is a source distribution for building the base Python package and should not be treated as the complete Git repository source. See the [Release Policy](../../legal/release_policy.md) for the official artifact boundaries.

The Windows base EXE is also intentionally limited: it contains the Qt shell and base CV, panorama, camera, and version workflows, but not the heavy classification or YOLO runtime. Use the source-setup link shown on those pages when those optional workflows are required.
