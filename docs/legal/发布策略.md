# 发布策略

[English](./release_policy.md) | [文档中心](../README.md) | [返回总 README](../../README.zh-CN.md)

Vision Workbench 是一个基于 AGPL-3.0 发布的开源学习项目。本文档定义 Git 源码、Python 构建包和模型发布资产的正式边界。

## 发布物类型

项目使用以下发布物。不同发布物服务于不同场景，不要求包含完全相同的目录。

| 发布物 | 用途 | 内容边界 |
| --- | --- | --- |
| Git 仓库、版本标签或对应源码归档 | 获取和审查完整项目源码 | 包含第一方源码、测试、正式文档、辅助脚本、项目元数据、许可文件、符合仓库策略的模型资产以及仓库内置的第三方源码 |
| Python sdist | 构建基础 Python 包 | 包含构建和识别 Python 包所需的源码、元数据、运行依赖声明和适用许可文件；不等同于完整 Git 仓库源码 |
| Python wheel | 安装基础应用和 Python 入口 | 包含第一方 Python 包、必要包内资源、入口和适用的许可文件；不要求包含测试、开发脚本、完整第三方仓库或大型模型 |
| GitHub Release Assets | 分发不适合进入 Git 或 wheel 的文件 | 包含正式 Python 包、Windows 基础 EXE 和更新清单，也可包含大型模型与可选离线资源 |

需要完整项目源码时，应使用对应版本的 Git tag 或源码归档，而不是将 wheel 或 Python sdist 视为完整仓库副本。

## 源码

完整源码发布包括 `src/`、`tests/`、`docs/`、`scripts/`、符合仓库策略的模型资产、项目元数据和适用的许可文件。

`third_party/yolo26_source/` 下的 YOLO26 源码作为学习和本地集成用途内置。该部分仍然遵守 Ultralytics 的 AGPL-3.0 许可和署名要求。

## 模型文件

只有单个文件不超过 100 MB 的模型文件才可按本策略公开。超过 100 MB 的文件不得直接提交到 Git 仓库。

大型文件建议通过以下方式提供：

- GUI 下载按钮
- 用户本地自行下载
- GitHub Release Assets
- 外部模型资源包
- 仓库维护者明确启用 Git LFS 后使用 Git LFS

公开发布前建议运行：

```bash
python scripts/check_release_assets.py
```

## 版本与更新契约

Vision Workbench 1.0.0 及后续正式版本必须同时提供更新清单所要求的精确版本 Python wheel 和稳定文件名 `Vision-Workbench-win-x64.exe`；EXE 内置身份必须与该次发布一致。只有最终资产生成后才能创建 `update-manifest.json`，并将该清单与相同 GitHub Release 一起发布。清单中的版本、tag、仓库、资产文件名、字节大小和 SHA-256 必须与不可变的正式资产完全一致。

桌面更新客户端只接受稳定正式版本。没有精确兼容文件名、有效正数大小、大小上限和 SHA-256 的资产不得一键安装，安装必须在 Qt 主进程退出后由独立进程执行。Python wheel 还必须包含一致的包元数据和内置发布身份；Windows EXE 必须在替换前通过版本与 Qt 构造自检。每次构建都要运行版本契约检查，正式发布模式只能在干净的发布 tag 上执行：

Python wheel 更新会刻意使用 `--no-deps`。因此清单会携带运行依赖契约的 SHA-256 指纹，其中包括基础依赖和可选运行依赖声明；只有它与当前运行版本一致时才开放一键安装。依赖契约发生变化时，必须按照发布说明执行手动安装，不能进行不完整升级。自包含的 Windows EXE 不依赖 Python 环境，因此不受这项仅针对 wheel 的兼容性门禁限制。

```bash
python scripts/check_version_contract.py
python scripts/check_version_contract.py --release
python scripts/generate_update_manifest.py --published-at <UTC_TIMESTAMP>
```

更新清单生成器会从干净且精确匹配的 tag 自动读取完整 commit SHA，不接受手工填写 commit 标识。

推送精确的稳定版 tag 后会启动 `Prepare release draft` 工作流。该流程会重新执行 tag 源码门禁，在原生 runner 上分别构建 Python 发布包和 Windows EXE，对已安装 wheel 与冻结 EXE 做冒烟自检，生成并解析更新清单，最后只创建未公开的 GitHub Release 草稿。人工检查资产并确认完整跨平台 CI 通过后，才能手动公开该草稿。

发布顺序必须如下，分支提交尚未通过审查和 CI 时不得提前创建 tag：

1. 在同一个待审查提交中同步版本、发布日期、更新日志、文档、依赖契约和测试。
2. 执行非发布模式版本契约、完整测试、文档链接和本地打包；推送分支提交并等待完整 CI 矩阵通过。
3. 在这个不再变化的提交上创建精确的附注 tag `v<major>.<minor>.<patch>`，然后只推送该 tag。
4. 等待 tag 工作流生成草稿；核对两个必需资产及其内置身份、清单哈希和大小、产物自检及 CI 状态后，再人工公开草稿。

## 构建产物

不要提交 `dist/` 目录下生成的 wheel 或源码压缩包。只有准备发布时再重新构建，并以对应版本的源码状态为准。

发布 wheel 应保持轻量。可选深度学习依赖、大型模型权重、测试、开发脚本和完整 vendored 仓库可以不进入 wheel；用户需要的附加内容应在 README、模块文档或 Release Assets 中明确说明。

## 许可文件

完整源码发布应保留：

- `LICENSE`
- `NOTICE`
- `THIRD_PARTY_NOTICES.md`
- `CITATION.cff`
- 第三方源码目录中保留原始许可文件

wheel 和 sdist 应包含与其实际分发内容相适用的许可与声明文件。未包含某个第三方组件的发布物，不应被描述为包含该组件的完整源码。
