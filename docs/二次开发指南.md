# 二次开发指南

[English](./adding_custom_features_README.md) | [文档中心](./README.md) | [返回 README](../README.md)

本文面向贡献者，说明 Vision Workbench 新能力应该加在哪里。普通运行用户请从根目录 [快速开始](../README.md#快速开始) 配置环境。

## 开发环境

```bash
conda create -n vision-dev python=3.11 -y
conda activate vision-dev
cd path/to/vision-workbench
pip install -e .[test]
pytest
```

只在开发对应模块时安装可选依赖：

```bash
python scripts/install_dependencies.py classification
python scripts/install_dependencies.py yolo26
```

## 分层模型

功能包按职责使用以下目录。除 `api/` 和 `application/` 等主要边界外，其余目录均按实际需要设置，不要求每个功能包机械包含全部层：

```text
api/             面向调用者的公共 Python API
application/     用例和工作流编排
configuration/   路径、参数和默认配置；按需设置
domain/          领域数据模型；按需设置
infrastructure/  文件、相机、模型和第三方库适配；按需设置
processing/      核心算法或模型处理；按需设置
ports/           协议和依赖边界；需要接口抽象时设置
```

`window/` 保存旧版 Tkinter 兼容/参考代码，不属于新功能包的标准分层。正式用户界面和新界面功能统一位于 Qt 桌面应用中。

公开桌面 UI 位于：

```text
src/vision_workbench/desktop/
src/vision_workbench/desktop/pages/
```

推荐调用链：

```text
Qt 页面 -> application -> processing / infrastructure -> application -> Qt 页面
```

算法和模型逻辑放在 `processing/`，外部资源适配放在 `infrastructure/`，业务流程放在 `application/`，用户可调用函数放在 `api/`。如果模块没有对应职责，可以省略相应目录，不创建空层。新的用户界面功能应修改 Qt desktop pages。除非承担明确的兼容责任，否则不要再给 `window/` 增加新的公开 GUI 入口。旧版界面的正式边界见[旧版界面维护策略](./旧版界面维护策略.md)。

## 依赖规则

基础依赖写入 `requirements.txt` 和 `pyproject.toml`。重依赖保持可选：

- 图像分类：`requirements-classification.txt`
- YOLO26 工作流：`requirements-yolo26.txt`

运行安装优先使用 `scripts/install_dependencies.py`，它可以根据机器自动选择 CUDA 或 CPU Torch。如果用户手动安装 `requirements-*.txt`，建议安装后执行 `python scripts/install_dependencies.py doctor`。

## 基础 CV 扩展

范围：新增图像处理效果、通道分析、直方图、形态学操作或几何变换。

主要文件：

```text
src/cv_basics/domain/models.py
src/cv_basics/processing/operations.py
src/cv_basics/api/facade.py
src/cv_basics/api/__init__.py
src/vision_workbench/desktop/pages/cv_basics_page.py
```

先在 domain 层添加效果名和参数，在 `processing/` 实现操作，暴露 API；需要界面时再添加 Qt 控件。

## 全景重构扩展

范围：拼接算法、配准策略、点选流程、融合或裁剪。

主要文件：

```text
src/panorama_reconstruction/processing/
src/panorama_reconstruction/application/reconstruction_service.py
src/panorama_reconstruction/api/facade.py
src/vision_workbench/desktop/pages/panorama_page.py
```

算法放在 `processing/`，文件和流程编排放在 `application/`，对外函数放在 `api/`，界面控件放在 Qt 页面。

注意：全景重构里的 `left` 和 `right` 是语义角色，不要按示例文件名推断空间位置。`left` 表示参考图，`right` 表示待拼接图，算法应把 `right` 变换到 `left` 坐标系。原因是历史示例资源文件名可能和空间语义相反；如果实现、UI 文案或测试只按文件名理解，会出现结果正确但界面语义反向的问题。

## 相机诊断扩展

范围：摄像头后端、读取模式、曝光/设置、截图、录屏和资源占用管理。

主要文件：

```text
src/camera_diagnostics/infrastructure/
src/camera_diagnostics/application/camera_service.py
src/camera_diagnostics/api/facade.py
src/vision_workbench/desktop/pages/camera_page.py
```

平台相关异常应在 infrastructure 层捕获，并通过 service 层返回清晰错误。

## 图像分类扩展

范围：新增 backbone、预训练权重、数据增强、指标、批量预测、checkpoint 加载或导出。

主要文件：

```text
src/image_classification/configuration/settings.py
src/image_classification/infrastructure/
src/image_classification/processing/classifier_backend.py
src/image_classification/application/classification_service.py
src/image_classification/api/facade.py
src/vision_workbench/desktop/pages/classification_page.py
```

权重校验和下载逻辑放在 infrastructure。训练/预测编排放在 service。API/service 契约稳定后再添加 Qt 控件。

## YOLO26 目标检测扩展

范围：检测模型、下载、摄像头推理、结果绘制、截图或运行参数。

主要文件：

```text
src/yolo26_detection/infrastructure/
src/yolo26_detection/application/detection_service.py
src/yolo26_detection/api/facade.py
src/vision_workbench/desktop/pages/yolo_detection_page.py
```

模型发现应先校验 `.pt` 文件，再把它们显示为可用模型。摄像头资源协调保留在 desktop 层，相机/模型适配保留在 infrastructure。

## YOLO26 分割扩展

范围：实例/语义分割任务、模型路由、结果渲染、下载或保存策略。

主要文件：

```text
src/yolo26_segmentation/infrastructure/
src/yolo26_segmentation/application/segmentation_service.py
src/yolo26_segmentation/api/facade.py
src/vision_workbench/desktop/pages/yolo_segmentation_page.py
```

`-seg` 和 `-sem` 模型规则必须按任务区分，本地 `.pt` 文件必须校验后再作为可用模型展示。

## YOLO26 训练扩展

范围：训练任务、参数、数据集校验、预训练权重规则、日志和输出管理。

主要文件：

```text
src/yolo26_training/infrastructure/
src/yolo26_training/application/training_service.py
src/yolo26_training/runner.py
src/yolo26_training/api/facade.py
src/vision_workbench/desktop/pages/yolo_training_page.py
```

数据集/模型校验放在 infrastructure 或 CLI runner。命令构造放在 service。Qt 页面负责收集参数、显示日志并启动 runner 进程。

## YOLO26 模型发现注意事项

本地项目模型、用户目录模型、内置默认模型、远端 manifest 和本地缓存都应通过 registry/repository 汇总后再交给 service/API。Qt 下拉框只读取本地聚合结果；只有启动刷新或用户点击“刷新模型列表”时才更新远端 manifest 缓存。原因是网络请求不稳定，直接绑定下拉框会造成界面卡顿、离线失败或难以复现的体验问题。

任务过滤规则属于开发约束：检测模型不要匹配 `-seg` 或 `-sem` 后缀，实例分割优先匹配 `-seg`，语义分割优先匹配 `-sem`。用户如何复制训练后的 `best.pt` 由训练模块说明文档维护；开发者在这里维护发现顺序、校验策略、缓存策略和测试。

新增模型来源或任务时，同步更新 `configuration/settings.py`、`infrastructure` 的 registry/repository、`application` 的刷新接口、`api/facade.py`、Qt 刷新动作和对应测试。缺一层时通常表现为“文件存在但下拉框不显示”，排查成本较高。

## Qt 文案注意事项

涉及文件选择和模型选择时，文案必须区分来源。从磁盘打开文件使用“选择本地模型文件”或“选择本地权重文件”；从已有列表选择使用“选择模型”；重新扫描或更新缓存使用“刷新模型列表”。原因是按钮文案会直接决定用户是否理解这是本地文件选择、下拉框选择，还是远端/本地缓存刷新。

如果文案承载流程语义，例如全景的“左图（参考）/右图（待拼接）”或模型按钮的“本地”，应在 Qt UI 测试里固定，避免后续重构把语义改回模糊状态。

## 文档归属注意事项

根目录 README 只放安装、启动和真正的快速开始。模块 README 放用户完成该模块工作流所需的信息。训练后的模型加载、下拉框扫描和 Python API 补充说明放在 YOLO26 训练模块目录。排错文档记录错误现象和处理办法。二次开发指南只记录结构约束、问题原因和扩展注意事项。

这样拆分可以降低重复内容过期或相互冲突的风险。需要由项目共同维护的结构约束和修改注意事项应写入正式二次开发或架构文档；具体调试过程、临时分析和维护者个人开发笔记可以保存在本地知识库中，不作为公开用户文档。用户需要执行的操作步骤继续放在对应模块文档中。

## 版本身份与更新改动

`pyproject.toml` 是正式版本号的唯一权威来源。运行时代码必须通过 `vision_workbench.versioning` 获取身份，不得重新加入数字形式的包级 `__version__` 常量，也不能从 EXE 文件名推导版本。每次正式版本变化必须同步 `release_info.json`、`CHANGELOG.md` 中英文两处当前版本标题、`CITATION.cff` 和两份根 README 的 wheel 示例；`scripts/check_version_contract.py` 会强制检查这组关系。

Python 一键更新刻意使用 `--no-deps`，因此基础或可选运行依赖声明发生变化时，依赖契约指纹也必须变化，并会阻止跨越该边界的一键 wheel 更新。修改任何依赖声明时，要同步内置指纹；版本契约检查会输出期望指纹并负责确认。Windows 正式资产保持稳定文件名 `Vision-Workbench-win-x64.exe`；其内置身份、Release tag、清单大小和 SHA-256 共同确定实际版本。

检查更新必须继续由用户显式触发，不能放入启动路径。新增更新来源或安装方式时，必须保留官方 HTTPS 主机校验、元数据和资产大小限制、wheel 内部身份检查、进程外安装，以及[发布策略](./legal/发布策略.md)规定的产物自检。

提交或发布前执行：

```bash
python -m compileall -q src tests scripts
python scripts/check_version_contract.py
python scripts/check_markdown_links.py
python -m pytest -q
python -m build
python scripts/cleanup_runtime.py
git status --short
```

公开 GUI 入口只有 `vision-workbench`。除非说明 legacy 兼容，否则用户文档不再推荐旧 `*/window/` 代码。
